# cratebox — 음반 통합 재고·정산 시스템 명세

중소 기획사·인디 레이블·소규모 음반 유통사를 위한 음반 도메인 특화 재고·정산 SaaS.
실물 음반 재고·유통 정산과 디지털 음원 정산을 하나의 시스템, 하나의 도메인 언어로 다룬다.

## 1. 사용자

| 역할 | 하는 일 |
|---|---|
| 유통사 (운영 주체) | 재고 입출고·반품·이동·실사, 거래처 관리, 거래처·기획사 정산 마감과 정산서 발행, 입금/지급 관리 |
| 기획사·레이블 | 자기 음반의 판매·재고·정산 내역 열람, 정산서 수신 (Phase ② 포털) |

멀티테넌시: 모든 도메인 테이블에 `org_id`. MVP는 단일 org(시드)로 운영하되 스키마·쿼리는 처음부터 org 스코프.

## 2. 단계별 범위

| | Phase ① (완료) | Phase ② (완료) | Phase ③ |
|---|---|---|---|
| 거래 모델 | 사입(유통사 소유)만 | + 위탁(기획사 소유 수탁) | — |
| 거래처 정산 기준 | 출고 기준(SELL_IN) + 판매분 기준(SELL_THROUGH) 모두 | — | — |
| 재고 | 사입입고·판매출고·진열출고·판매보고·반품·회수·창고이동·실사·기초재고 | + 수탁입고/반납, 소유권 차원 활성화 | — |
| 정산 | 월 마감 → 거래처 정산서 + 기획사 매입 정산서, 입금/지급, 마감 후 반품(이월/소급) | + 위탁 정산서, MG 선차감, 기획사 포털 | + 음원 정산 업로드→정규화→분배→아티스트 정산서 |
| 상품 | 앨범→버전→SKU | — | + 트랙, 분배율(splits) |
| 리포트 | 재고현황(위치별), 초동, 거래처/기획사 잔액, 원장 조회 | 위탁 재고 현황 | 실물+음원 통합 정산 뷰 |

## 3. 도메인 규칙 (Phase ①)

### 3.1 상품
- `앨범 → 버전(A/B/C, 한정반) → SKU`. SKU가 판매·재고 단위이며 바코드(org 내 유일), 소비자가(정가), 발매일(버전 상속)을 가진다.
- 랜덤 포토카드 등 버전 내부 구성물은 재고 단위로 관리하지 않는다(버전=SKU가 일반형, 세트 상품은 별도 SKU).

### 3.2 거래처와 정산 기준
- `party` 하나로 기획사(LABEL)와 거래처(RETAILER)를 통합 — 정산 원장이 상대방을 단일 FK로 참조하기 위함.
- RETAILER는 `settlement_basis`를 가진다:
  - **SELL_IN (출고 기준)**: 출고 시점에 채권 발생, 반품 시 차감. `판매출고`/`거래처반품` 문서 사용.
  - **SELL_THROUGH (판매분 기준)**: 출고(진열)는 재고 이동일 뿐, 거래처의 **판매보고** 시점에 채권 발생. 미판매분은 `회수`로 돌아온다. 매장에 깔린 재고는 거래처 위치(location)의 자사 소유 재고로 원장에 남는다.
- 공급가: 거래처별 기본 공급률(소비자가 대비, `default_supply_rate`) → 문서 라인에서 단가 수정 가능. 원장에는 확정 시점 단가가 스냅숏된다.

### 3.3 금액·부가세
- 통화 KRW, 금액은 `bigint`(원). 단가는 공급가액(VAT 별도) 기준.
- 부가세 10%, 라인별 계산, 원 미만 절사(음수 라인은 절대값 절사 후 부호 적용).
- 잔액·정산서 합계는 VAT 포함 금액 기준. 입금/지급은 VAT 구분 없는 현금 금액.

### 3.4 문서와 전기(posting)
- 사용자는 **문서**를 작성한다(DRAFT). **확정(POST)** 시 원장 엔트리가 생성된다. 원장은 append-only — DB 트리거로 UPDATE/DELETE 차단.
- 수정·취소는 **역분개**: 확정 문서에 역분개를 걸면 부호 반전 엔트리(원본 `entry_type` 유지, `reversal_of_id` 표시)가 생성되고 원 문서는 REVERSED 상태가 된다. 필요 시 새 문서를 다시 작성.
- 문서 타입: `PURCHASE_IN`(사입입고) `PURCHASE_RETURN`(매입반품) `SALE_OUT`(판매출고) `CONSIGN_PLACE`(진열출고) `SALES_REPORT`(판매보고) `CUSTOMER_RETURN`(거래처반품) `CONSIGN_RECALL`(회수) `TRANSFER`(창고이동) `ADJUST`(실사조정) `OPENING`(기초재고).
- `SALES_REPORT`는 음수 라인 허용(소비자 환불 = 판매 취소 보고).
- 출고류는 확정 시점에 해당 위치·SKU 잔량 부족이면 거부(음수 재고 불가). 실사조정만 signed 수량.

### 3.5 정산 마감과 정산서
- 정산 주기: 월력(1일~말일). `settlement_period(org, year_month)`는 순차적으로만 마감(이전 달 마감 없이는 당월 마감 불가).
- **마감 = 미배정(`settlement_period_id IS NULL`) 정산 엔트리 중 `occurred_on ≤ 월말`인 것을 해당 기간에 배정(도장)하고 상대방별 정산서를 생성**하는 것.
- 정산서 2종: 거래처 정산서(RETAILER), 기획사 매입 정산서(LABEL_PURCHASE). 형식: `전기 이월 잔액 + 당기 발생(출고/판매−반품) − 입금/지급 = 당기 말 잔액` + SKU별 명세.
- 정산서는 버전을 가진 불변 스냅숏. 재발행 시 version+1, 원본 보존.

### 3.6 마감 후 반품 (명시 설계)
- **차기 이월(기본)**: 마감 후 확정된 반품의 정산 엔트리는 미배정 상태이므로 다음 마감에 자연 포함된다. 별도 로직 없음.
- **소급 정정**: 반품 문서 확정 시 마감된 기간을 명시 지정 → 엔트리가 그 기간에 배정되고 해당 정산서를 version+1로 재발행(차액 확인 가능). 연쇄 재발행 방지를 위해 **직전 마감 기간까지만** 소급 허용.
- 거래처별 기본 모드(`late_return_mode`: CARRY_FORWARD | RESTATE)를 두되, 문서 단위로 재량 선택 가능.

### 3.7 초동
- 버전 발매일 포함 7일간의 순출하량: `SALE_OUT + SALES_REPORT − CUSTOMER_RETURN`(수량 기준, 역분개 자동 반영). 화면에 "유통사 출고 기준"임을 명시(한터 등 POS 집계와 다름).

### 3.8 입금/지급
- `payment`: 거래처 입금(IN, 채권 감소), 기획사 지급(OUT, 채무 감소). 정산 원장에 엔트리로 전기되어 잔액·정산서에 반영. 취소는 역분개.

## 3+. 도메인 규칙 (Phase ② 위탁)

### 3.9 거래 계약과 위탁 재고
- `trade_agreement`는 **앨범 단위**(기획사는 `album.label_party_id`로 결정): 사입(PURCHASE) 또는 위탁(CONSIGNMENT, 수수료율 필수). 위탁 재고를 다루는 모든 문서의 전제 조건.
- 위탁 재고 = 재고 원장의 **소유 차원**: 문서 라인의 `owner_party_id`(null=자사)가 어느 풀에서 빼고 더할지를 정한다. 소유자는 반드시 해당 앨범의 기획사. 같은 SKU라도 자사 풀과 위탁 풀은 분리 집계·분리 잠금(한쪽 부족 시 거부).
- `수탁입고 CONSIGN_IN`(기획사→창고, owner=기획사)과 `위탁반납 RETURN_TO_OWNER`(창고→기획사)는 재고 이동일 뿐 정산이 없다. 사입입고/매입반품 라인에는 위탁 소유 지정 불가.

### 3.10 위탁 판매의 이중 전기
- 위탁 라인의 판매(판매출고/판매보고)·반품(거래처반품)은 **정산 엔트리 2행**: 거래처 행은 사입과 동일(±공급가액+VAT), 기획사 행은 `CONSIGN_SALE`/`CONSIGN_RETURN`으로 반대 부호.
- **기획사 몫 = 원 미만 절사(|공급가액| × (1 − 수수료율))**, 나머지가 유통사 수수료(절사분은 유통사에 유리). VAT는 기획사 몫에 대해 별도 계산(동일 절사 규칙). 기획사 엔트리에는 단가를 기록하지 않는다(수수료 차감 후 금액이라 qty×단가가 성립하지 않음).
- 한 판매 건의 정산 엔트리 합 = 유통사 마진(수수료+VAT 차액)이 원장에서 그대로 읽힌다.

### 3.11 MG 선급금
- `advance`(기획사×앨범, 지급액)는 **정산 잔액 밖에서** 관리한다 — MG 지급을 별도 지급(payment)으로 기록하지 않는다.
- **마감 시 자동 회수**: 해당 앨범의 당기 위탁 정산액(CONSIGN_SALE/RETURN 합, 지급할 금액)에서 선급금 잔여만큼 `ADVANCE_RECOUP`(양수) 엔트리로 차감. 오래된 선급금부터, 같은 기획사×앨범 다건은 순차. 회수 엔트리는 생성 즉시 해당 기간에 도장된다.
- 정산서에 `advance_total`(MG 차감)로 표시. 기말 잔액 = 이월 + 당기 + 입금/지급 + MG 차감.
- **명시된 단순화**: 마감 후 그 기간으로 소급 정정이 들어와도 회수는 재계산하지 않는다(잔액 합계는 항상 정합 — MG가 약간 일찍 회수될 뿐). 회수가 시작된 선급금은 삭제 불가.

### 3.12 기획사 포털
- 역할 2종: `ADMIN`(유통사 운영자, 전체), `LABEL`(기획사, `party_id` 스코프 읽기전용).
- 운영자가 기획사당 1개 포털 계정을 발급. LABEL 로그인 시 `/api/portal/*`만 접근 가능(운영 API 403), 포털은 자기 위탁 재고(창고/매장), 자기 정산서(전 버전), 자기 정산 원장만 노출.

### 3.13 현장판매 (②+)
- `DIRECT_SALE`: 행사·팝업에서 소비자 직판. **상대방 없음** — 현장 수금이므로 거래처 채권을 만들지 않는다(명시 단순화). 재고는 출고 창고에서 차감, **위탁 라인은 기획사몫(CONSIGN_SALE)이 그대로 전기**되어 마감·MG 회수에 자연 합류.
- 모바일 현장판매 화면: 바코드 입력(블루투스 스캐너·수동) + 카메라 스캔. 인식은 zxing-wasm 폴리필(로컬 번들)이라 **iOS 사파리 포함 전 브라우저 동일 동작**. 단, 브라우저 카메라는 보안 컨텍스트에서만 열리므로 폰에서는 `https://<서버IP>:8443`(자체서명 추가 커넥터)으로 접속. 같은 품목 재스캔 = 수량 +1. 기본 단가 = 정가/1.1 역산(원장 단가는 항상 VAT 별도 공급가).

### 3.14 발행사 정보·계정 관리 (②+)
- org에 회사 정보(사업자번호·대표자·주소·연락처·이메일) 저장, 정산서 상세(운영·포털)와 인쇄물 머리글에 발행사로 표기.
- 비밀번호: 본인 변경(`POST /api/auth/password`, 현재 비밀번호 확인, ADMIN·LABEL 공용) + 운영자의 포털 계정 재설정(`PUT /api/parties/{id}/portal-user/password`).

## 4. 기술 스택 (확정)

| 영역 | 선택 | 근거 |
|---|---|---|
| 백엔드 | Java 21, Spring Boot(최신 안정), Spring Data JDBC + JdbcTemplate, Flyway | 원장 패턴은 명시적 SQL 제어가 맞다. JPA 연관관계·더티체킹은 append-only 모델과 상성이 나쁨 |
| DB | PostgreSQL 16 (Docker) | 정산 정합성·이력. 트리거로 원장 불변성 강제 |
| 프런트 | React + TypeScript + Vite + TanStack Query/Table + Tailwind | 데이터 테이블 중심 어드민, 반응형 비용 낮음 |
| 배포 형태 | Spring Boot가 프런트 정적 번들 서빙, 단일 배포물 | 운영 단순화 |
| 인증 | 세션 로그인, 시드 관리자 계정 + 역할(ADMIN/LABEL, ②에서 도입) | MVP 최소 |

아키텍처: 모듈러 모놀리스. 패키지 = `catalog`(상품) / `party` / `inventory`(문서·재고원장) / `settlement`(정산원장·마감·정산서) / `auth`.

## 5. 자체 검증 계획

빌드 중 반복 실행하는 테스트 스위트(실 Postgres 대상):

| # | 검증 항목 |
|---|---|
| T1 | 문서 타입별 확정 → 기대 원장 엔트리·잔량 (전기 매트릭스 전 행) |
| T2 | 잔량 부족 출고 거부, DRAFT 아닌 문서 확정 거부 |
| T3 | 역분개 → 잔량 원복, (원본+역분개) 합계 0, 집계 리포트 자동 반영 |
| T4 | 골든 시나리오: 한 달치 운영(입고→출고→판매보고→반품→입금→마감)을 API로 수행, 손으로 계산한 정산서 숫자와 전액 대조 |
| T5 | 마감 후 반품: 이월 → 차기 정산서 포함 / 소급 → 해당 기간 정산서 v2 + 차액 |
| T6 | 판매분 기준 흐름: 진열→판매보고→회수, 거래처 위치 잔량 정확성 |
| T7 | 불변식: 모든 (SKU,위치,소유) 키에 대해 Σ원장 == 잔량 캐시 — 각 시나리오 종료 시 검사 |
| T8 | 원장 UPDATE/DELETE 시 DB 트리거가 거부 |
| T9 (②) | 위탁 이중 전기 손계산 대조, 소유 풀 분리(자사 재고가 있어도 위탁 풀 부족이면 거부), 계약·소유자 검증 거부 |
| T10 (②) | MG 앨범 단위 회수: 위탁분만 차감(같은 기획사의 사입 채무는 잔존), 잔여 추적, 수수료 절사, 위탁 역분개 원복 |
| T11 (②) | 포털 스코프: LABEL 계정은 자기 재고·정산서만(남의 정산서 404), 운영 API 403, 운영자는 포털 403 |
